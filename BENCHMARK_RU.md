# Бенчмарк: ModDict vs обычный Python dict

Все результаты измерены на **100 000 структурированных строках** (или 1 000 000 скалярных записей).
Абсолютные значения зависят от железа; соотношения стабильны на разных платформах.

Схема строки: `age int | score float | name str | active bool | tags list | joined date`
`| meta.level int | meta.badge str | meta.score_v2 float`
`| meta.details.region str | meta.details.rank int`
(3 уровня вложенности, 11 листовых полей на строку)

> **Архитектура:** строки хранятся как `PyObject*` — тот же dict, который передал вызывающий код, с `Py_INCREF`. Глубокого копирования нет. Доступ к полям через Python chaining на хранимой ссылке: `mn[key]["field"]`. Записи через `mn[key]["field"] = val` перехватываются **RowProxy**, который автоматически синхронизирует FieldIndex.

---

## Скалярное хранилище (1 000 000 int-записей)

| Операция | dict | ModDict | соотношение |
|----------|------|---------|-------------|
| `d[key]` vs `m[key]` | 187ms | 190ms | **~равно** |
| `d.get(key)` vs `m.get(key)` | 199ms | 244ms | 0.81× медленнее |
| `key in d` vs `key in m` | 170ms | 183ms | 0.93× медленнее |
| `d[key] = val` vs `m[key] = val` | 265ms | 267ms | **~равно** |
| массовая вставка 1M | 244ms | 275ms | 0.89× медленнее |

Для чисто скалярных задач dict имеет небольшое преимущество — ModDict добавляет overhead на refcount и проверку индекса при вставке. Если данные полностью скалярны — используйте dict.

---

## Структурированные строки — чтение (100 000 строк)

| Операция | dict | ModDict | соотношение |
|----------|------|---------|-------------|
| `d[key]` vs `mn[key]` (полная строка) | 9ms | 14ms | 0.61× медленнее |
| `d[key]["age"]` vs `mn[key]["age"]` | 14ms | 18ms | 0.79× медленнее |
| `d[k]["meta"]["details"]["rank"]` vs то же на mn | 29ms | 32ms | 0.90× медленнее |

`mn[key]` возвращает **RowProxy** когда активны field-индексы — тонкую обёртку, которая перехватывает записи для поддержания индексов актуальными. Доступ на чтение прозрачно делегируется хранимому dict.

---

## Структурированные строки — запись (100 000 строк)

| Операция | dict | ModDict | соотношение |
|----------|------|---------|-------------|
| массовая вставка 100k строк | 25ms | 27ms | **~равно** |
| `d[k]["age"] = 99` vs `mn[k]["age"] = 99` | 19ms | 26ms | 0.75× медленнее |
| `d[k]["meta"]["details"]["rank"] = 99` vs то же на mn | 31ms | 36ms | 0.85× медленнее |

`mn[k]["age"] = 99` проходит через RowProxy, который автоматически обновляет FieldIndex за O(1). Накладные расходы по сравнению с dict невелики и дают консистентность индексов бесплатно.

## Фильтрация (100 000 строк)

Индекс строится автоматически при **первом** вызове `filter()` и переиспользуется далее.

| Операция | dict | ModDict | соотношение |
|----------|------|---------|-------------|
| сканирование `age >= 40` vs `filter("age").gte(40)` — 1-й вызов | 20ms | 28ms | 0.70× |
| то же — 2-й+ вызов (переиспользование индекса) | 20ms | 26ms | 0.78× |
| сканирование `active == True` vs `filter("active").eq(True)` — 1-й вызов | 17ms | 12ms | **1.45×** быстрее |
| то же — 2-й+ вызов | 17ms | 10ms | **1.63×** быстрее |

Поиск по булевому индексу — O(1). Числовой диапазонный скан (`age >= 40`) приближается к скорости линейного сканирования при высокой плотности совпадений (~60%); ModDict выигрывает при высокой селективности и на больших датасетах.

## Сортировка, группировка, проекция (100 000 строк)

**Здесь индексированная структура ModDict даёт наибольший выигрыш.**

| Операция | dict | ModDict | соотношение |
|----------|------|---------|-------------|
| `sorted(d, key=lambda k: d[k]["age"])` vs `mn.sort_by("age")` | 52ms | 15ms | **3.54×** быстрее |
| `groupby active` (2 группы) | 43ms | 19ms | **2.27×** быстрее |
| `groupby age` (~63 уникальных значения) | 1033ms | 19ms | **55×** быстрее |
| `{k: {f: d[k][f] for f in fields} for k in d}` vs `mn.select([...])` | 201ms | 93ms | **2.15×** быстрее |

Индекс строится автоматически при первом вызове `sort_by` / `group_by` и переиспользуется. Выигрыш в group_by растёт с кардинальностью — dict-подход масштабируется O(n·групп), ModDict — O(n).

## Обновление / слияние (100 000 строк)

| Операция | dict | ModDict | соотношение |
|----------|------|---------|-------------|
| key→key `*,*` (обновление только существующих) | 49ms | 38ms | **1.29×** быстрее |
| key→key `?,?` (обновление + вставка новых ключей) | — | 37ms | — |
| обновление по значению поля (`user_id` → `*`) | 60ms | 26ms | **2.29×** быстрее |

`mn.update(other, "*", "*")` — обновляет только ключи, уже присутствующие в self.  
`mn.update(other, "?", "?")` — то же, плюс **вставляет** ключи из `other`, отсутствующие в self.  
Оба варианта не пересобирают индексы для незатронутых полей.

## Сериализация (100 000 строк)

Поддерживаемые типы: `None` `bool` `int` `float` `str` `bytes` `list` `set` `frozenset` `dict`
`datetime.datetime` `datetime.date` `datetime.time`
`decimal.Decimal` `pathlib.Path` `pathlib.PurePosixPath` `pathlib.PureWindowsPath`
плюс пользовательские типы через `md.register_converter(MyType, encoder)`

| формат | сериализация | десериализация | размер |
|--------|-------------|----------------|--------|
| **ModDict binary** | 260ms | 410ms | 32.5 МБ |
| json | 252ms | 341ms | 23.7 МБ |
| pickle | 756ms | 488ms | 25.4 МБ |

ModDict binary поддерживает полный набор Python-типов (date, bytes, Decimal, Path, …) без кастомного энкодера — в отличие от json. По сравнению с pickle: **в 2.9× быстрее сериализация, в 1.2× быстрее десериализация**.

## Итерация (100 000 строк)

| dict | ModDict | соотношение |
|------|---------|-------------|
| `for k in d` 2ms | `for k in mn` 3ms | 0.67× медленнее |

ModDict сканирует плоскую хэш-таблицу; dict использует компактный split-index дизайн. Разрыв сократился после того, как размер `OuterEntry` уменьшился до 24 байт — ровно как у CPython dict entry. Если итерация — узкое место, соберите ключи один раз через `mn.keys()`.

## Новое в этом релизе (100 000 строк)

| Операция | аналог dict | dict | ModDict | соотношение |
|----------|-------------|------|---------|-------------|
| `ModDict.from_rows(rows, key="id")` | `{r["id"]: r for r in rows}` | 17ms | 86ms | 0.20× медленнее |
| `mn.copy()` (глубокое копирование) | `copy.deepcopy(d)` | 1313ms | 169ms | **7.79×** быстрее |
| `mn.at(0)` (первый по порядку вставки) | `list[0]` | ~0ms | ~0ms | **2.37×** быстрее |
| `mn.at(-1)` (последний по порядку вставки) | `list[-1]` | ~0ms | ~0ms | **2.27×** быстрее |

`from_rows` медленнее dict comprehension, потому что индексирует каждую строку при вставке. `copy()` выигрывает в **7.8×** у `deepcopy` — ModDict рекурсирует только по dict-значениям и пропускает overhead общего Python-копировщика.

## Wildcard-фильтр (1 000 внешних × 100 внутренних строк)

Датасет: `{group_key: {row_key: {user_id, score, ...}}}` — двухуровневое вложение.
`filter()` на нетерминальном wildcard возвращает **pruned**-результат: в итоге
остаются только совпавшие inner-ключи, поэтому dict-сравнения ниже строят такую
же pruned-структуру (а не проверку через `any()`) — для честного сравнения.

| Операция | аналог dict | dict | ModDict 1й вызов | ModDict 2й+ |
|----------|-------------|------|-----------------|-------------|
| `filter("?.user_id").eq(5)` | pruned `{gk: {rk: rv for rk,rv in gv.items() if rv["user_id"]==5}}` | 9ms | 1ms (**10.8×**) | 1ms (**15.7×**) |
| `filter("?").eq("r1")` — терминальный, проверка ключа | pruned `{gk: {"r1": gv["r1"]}}` для совпавших групп | ~0ms | ~0ms (**1.5×**) | ~0ms (**1.6×**) |
| `filter("g1.?.user_id").eq(5)` — anchor | `{rk: rv for rk,rv in d["g1"].items() if rv["user_id"]==5}` | ~0ms | ~0ms (**2.1×**) | ~0ms (**3.6×**) |
| `filter("?.?.status").eq("Active")` — 2 уровня wildcard | pruned 3-уровневый nested dict comprehension | 12ms | 7ms (**1.6×**) | 7ms (**1.7×**) |
| `.eq(5, returns="rows_here")` | `[r for gv in d.values() for r in gv.values() if r["user_id"]==5]` | 7ms | 10ms | 10ms |
| `.eq(5, returns="values", value_field="score")` | `[r["score"] for gv ...]` | 7ms | 10ms | 10ms |

**Семантика путей:**
- `"?.user_id"` — `?` пропускает один уровень ключей, фильтрует по `user_id` в значении. Строит индекс → **15.7× быстрее** на повторных вызовах.
- `"?"` (терминальный) — проверяет, равен ли **ключ** указанному значению. Полезно для «есть ли такой ключ в группе?»
- `"g1.?.user_id"` — anchor: первый сегмент является конкретным внешним ключом, сканирование ограничено одной строкой.
- `"?.?.status"` — один `?` на уровень вложенности (нет неявного пропуска нескольких уровней — для более глубокой структуры пишите `?` явно на каждый уровень). Использует тот же индекс-based механизм восстановления, что и одиночный `?`, без пересканирования.
- `returns="rows_here"` — возвращает внутренние dict на уровне совпадения (линейный обход, без индекса).
- `returns="values", value_field="score"` — извлекает одно поле из каждого совпадающего внутреннего dict.

Нетерминальный wildcard EQ (любое число уровней `?`) и терминальный `?`
восстанавливаются напрямую из индекса — без пересканирования строки. `ne()`
и range-операции (`lt`/`gt`/...) на wildcard-путях пока не имеют быстрого пути
через индекс и падают на полный скан при каждом вызове.

## to_dict / dumps / loads (100 000 строк)

| Операция | Примечание | Время |
|----------|-----------|-------|
| `mn.to_dict()` | обычный dict, минует RowProxy | 23ms |
| `dict(mn)` | keys()+getitem, может вернуть RowProxy при наличии индекса | 24ms (0.94×, медленнее) |
| `md.dumps(plain_dict)` | обобщённый формат одного значения | 169ms |
| `md.dumps(mn)` | нативный контейнерный формат ModDict (тот же, что `mn.serialize()`) | 271ms |
| `md.loads(dumps(dict))` | → `dict` | 488ms |
| `md.loads(dumps(mn))` | → `ModDict` | 576ms |

`md.dumps()`/`md.loads()` — модульные функции для сериализации **любого**
поддерживаемого объекта, не только целиком `ModDict`. `ModDict` возвращается
обратно как `ModDict`, всё остальное — как есть. Неявной конвертации
`ModDict` → `dict` нет — вызови `mn.to_dict()` сначала, если нужен именно
plain dict.

## Связи: link() / follow() / '->'-фильтр (100 000 заказов × 10 000 клиентов)

Двухтабличный `ModDict`: `orders: {pk: {customer_id}}`, `customers: {pk: {name}}`,
связь объявлена через `mn.link("orders.?.customer_id", "customers.?")`.

| Операция | dict | ModDict | соотношение |
|----------|------|---------|-------------|
| `link()` объявление + валидация (100k заказов) | — разовая стоимость, аналога в dict нет | — | 51ms | — |
| полный join: `{ok: custs[o["customer_id"]] for ok,o in orders.items()}` vs `mn.follow(...)` | 15ms | 12ms | **1.29×** быстрее |
| join+фильтр по имени клиента (`customers[o["customer_id"]]["name"]==target`) vs `filter("orders.?.customer_id->name").eq(target)` — 1-й вызов | 17453мкс | 13.7мкс | **1274×** быстрее |
| то же — 2-й+ вызов (индекс на целевом поле тоже переиспользуется) | 17453мкс | 3.5мкс | **5056×** быстрее |

Разовая стоимость `link()` — это валидация, что каждая существующая строка
резолвится в реальную целевую строку — та же категория затрат, что и
построение `FieldIndex`, оплачивается один раз, сколько бы раз потом ни
вызывались `follow()`/`filter()`.

`follow()` всё равно обходит каждую строку источника, так же как ручной
join — выигрыш здесь обычная разница в overhead на строку, не другой
алгоритм.

А вот `filter("...->name").eq(target)` — это **другой алгоритм**: он вообще
не сканирует таблицу orders. Сначала резолвится "какие клиенты подходят"
(индексированный поиск по 10 000 клиентов), затем через уже построенный
`link()`-ом индекс делается обратный поиск "какие заказы ссылаются на этих
клиентов" — O(совпадений), а не O(размера таблицы orders). При 1 целевом
клиенте из 10 000 (≈10 совпадающих заказов из 100 000) это касается порядка
10 строк независимо от размера таблицы — замеры выше усреднены по 500
повторам (не единичный шумный сэмпл, округлённый до целых миллисекунд), и
подтверждают: **стоимость быстрого пути зависит от числа совпадений, а не от
размера таблицы orders**.

## Курсоры (реактивные виды)

Ближайший аналог для отсортированного, реагирующего на мутации курсора —
обычный Python `list`, поддерживаемый отсортированным вручную: `bisect.insort`
для одиночных вставок, `append`+`sort()` для батчей. Именно так выглядит
самодельная GUI-табличная модель без курсора (см. `RecordTableModel` из
`app_copy` — то, что курсор заменяет) — каждая строка ниже измерена именно
против него, не в вакууме.

| Операция | list | cursor | отношение |
|-----------|------|--------|-------|
| создание `cursor()` (любой размер anchor'а, 1k–50k строк) | — | 1.7–2.0мкс | константа, таблица не сканируется |
| `bisect.insort` vs `insert()` (1 000 строк, sort активен) | 2.1мкс | 3.5мкс | 0.61× (в 1.7× медленнее) |
| `bisect.insort` vs `insert()` (10 000 строк) | 4.8мкс | 8.2мкс | 0.59× (в 1.71× медленнее) |
| `bisect.insort` vs `insert()` (50 000 строк) | 13.8мкс | 19.7мкс | 0.70× (в 1.43× медленнее) |
| `bisect.insort` × 5 000 в цикле vs `append`+`sort()` разом (list, его собственная batch-стратегия) | 9мс | 2мс | в 5.57× быстрее |
| `append`+`sort()` (list) vs `insert_batch()` (cursor), оба batch-style | 2мс | 1мс | **в 2.59×** быстрее — batch курсора обгоняет даже лучшую стратегию самого list |
| `insert()` × 5 000 в цикле vs `insert_batch()` разом | 250мс | 1мс | **в 401×** быстрее |
| `sorted(list)` vs `set_sort()` bootstrap (1k / 10k / 50k строк) | 0 / 2 / 10мс | 0 / 6 / 39мс | ~в 2.8–3.9× медленнее — см. ниже |
| list comprehension vs `set_filter()` bootstrap (1k / 10k / 50k строк) | 0 / 1 / 3мс | 0 / 2 / 15мс | ~в 3.7–5× медленнее — см. ниже |
| `list.sort(key=group)` vs `set_group()` bootstrap (1k / 10k / 50k строк) | 0 / 1 / 8мс | 0 / 4 / 26мс | ~в 2.1–3.2× медленнее — см. ниже |
| `insert()` с 0 / 1 / 5 / 20 зарегистрированными сиблинг-курсорами (sort активен) | — | 1.8мкс / 1.1мс / 5.5мс / 22.3мс | растёт линейно с числом живых сиблингов |
| поштучный `mn[k]=v` × 20 000 в проиндексированный `ModDict` vs `mn.update()` (батчевый пересчёт `FieldIndex`) | 237мс | 62мс | **в 3.83×** быстрее |

**`insert()` укладывается в ~1.4–1.7× от голого `bisect.insort` на Python
list** — близко к теоретическому пределу с учётом того, что курсор делает
сверх списка: пишет напрямую в реальный dict родителя, поддерживает hook
поле-индекса, проверяет зарегистрированных сиблинг-курсоров. До этого
результата дошли за два раунда фиксов: bisect fast-path (`O(log n)`-поиск +
чисто указательный сдвиг `vector::insert` вместо `O(n log n)`-полной
пересортировки на каждую одиночную вставку), а затем обнаружили, что и сам
fast-path всё ещё честно перечислял каждую сдвинутую строку как явную пару
`(old_index, new_index)` — дёшево в C++, но каждая пара означала аллокацию
2 Python-int + tuple, что на масштабе и доминировало (~25 000 пар на одну
вставку в таблицу из 50 000 строк). `insert()`/`delete()`/`insert_batch()`
теперь репортят только те строки, что реально записываются, а не каждого
соседа, которого GUI-шный `beginInsertRows`/`beginRemoveRows` и так
перенумерует неявно — `update_row()` единственное исключение, он возвращает
оба индекса, потому что сдвиг из-за смены поля — это `beginMoveRows`, и
"откуда" GUI сам вывести не может.

**Bootstrap `set_sort()` изначально был в 20–33× медленнее `list.sort()`,
теперь — ~в 2.8–3.9×.** Причина была не в "C++ vs Python" — компаратор
сортировки заново извлекал и конвертировал значение поля каждой строки на
*каждом* попарном сравнении (`O(n log n)` извлечений для сортировки
50 000 строк — около 1.5 млн обращений к dict/полю ради тех же 50 000
значений). Собственный `list.sort(key=...)` в Python избегает этого именно
через decorate-sort-undecorate: ключ каждого элемента считается один раз,
а сортируются уже дешёвые предвычисленные ключи. `rebuild_sort_index()`
теперь делает то же самое — предвычисляет `ModValue` sort/group-поля всех
строк за один `O(n)` проход, а затем сортирует по предвычисленным
значениям. Оставшийся разрыв (~3–4×) — реальные, неустранимые накладные
расходы сравнения `ModValue`/`PyObject`, за которые голый Python
`sort(key=...)` никогда не платит — это больше не устранимая алгоритмическая
разница.

**Bootstrap `set_filter()` в ~3.7–5× медленнее list comprehension** —
вызов пользовательского предиката на каждую строку (`PyObject_CallOneArg`
обратно в Python) плюс вставка в hash-map на каждую прошедшую строку
неизбежно тяжелее, чем `if` на уровне интерпретатора внутри comprehension,
и в отличие от sort здесь нет избыточности по сравнениям, которую можно
устранить — каждая строка в любом случае обрабатывается ровно один раз.
Тот же профиль стоимости "редкий, явный reconfigure", что и у `set_sort()`.

**Bootstrap `set_group()` в ~2.1–3.2× медленнее `list.sort(key=group)`** —
он использует тот же путь `rebuild_sort_index()` "предвычислить-затем-
отсортировать", что и `set_sort()` (значение group идёт первым в
компараторе, sort — тай-брейкером), поэтому получил ту же оптимизацию;
разрыв здесь уже, чем у чистого `set_sort()`, потому что list-компаратор
в этом бенчмарке тоже делает реальную работу (сравнение строк на каждый
элемент), а не тривиальную операцию.

**Стоимость нотификации сиблингов линейна по числу живых курсоров** на
одном anchor'е с активным sort — каждый получает свой полный ресинк, когда
*другой* курсор мутирует общие данные (и это тоже ускорилось благодаря фиксу
`set_sort()` выше, так как ресинк использует тот же `rebuild_sort_index()`).
Это осознанный компромисс, не баг: GUI на курсорах обычно держит горстку
видов на одну таблицу, не сотни.

## Итого

| Задача | Рекомендация |
|--------|-------------|
| Только скаляры | dict — ModDict добавляет ненужный overhead |
| Чтение полей вложенного dict | dict — незначительно быстрее |
| Массовая вставка строк | **~равно** |
| Сортировка по полю | **ModDict** — **3.5×+** быстрее (растёт с размером датасета) |
| Группировка по полю | **ModDict** — до **55×** быстрее |
| Фильтрация (булевые поля) | **ModDict** — **1.6×** быстрее |
| Фильтрация (числовой диапазон, плотный) | ~равно; ModDict выигрывает при высокой селективности |
| Проекция полей (select) | **ModDict** — **2.1×** быстрее |
| Обновление / merge по ключу | **ModDict** — **1.3–2.3×** быстрее |
| Глубокое копирование | **ModDict** — **7.8×** быстрее `deepcopy` |
| Доступ по индексу `at(i)` | **ModDict** — O(1) через вектор порядка вставки |
| Сериализация | ModDict если нужны date/bytes/Decimal; json для минимального размера |
| Asyncio shared cache | **ModDict** — zero-copy `PyObject*`, без GC-давления при чтении |
| Инициализация из dict / ModDict / Mapping | dict быстрее; копия ModDict ~2× медленнее (полная переиндексация) |
| Wildcard-фильтр (`?.field`, любое число уровней) | **ModDict** — до **15.7×** быстрее на повторных вызовах |
| `to_dict()` vs `dict(mn)` | **~равно**; `to_dict()` всегда минует RowProxy |
| `dumps`/`loads` (любой объект, не только ModDict) | см. отдельный раздел — сопоставимо с `serialize()`/`deserialize()` |
| `follow()` (объявленная связь, резолв каждой строки-источника) | **ModDict** — **1.4×** быстрее ручного join |
| `filter("...->field").eq(x)` (JOIN в WHERE) | **ModDict** — O(совпадений) через индекс, не O(размера таблицы) — **1000×+** быстрее при низкой селективности |
| Курсорный `insert()` vs `bisect.insort` на отсортированном list | **~равно** — в пределах 1.4–1.7× на 1k–50k строк |
| Курсорный `insert_batch()` vs list `append`+`sort()` | **cursor** — **в 2.59×** быстрее |
| Bootstrap `set_sort()` курсора vs `list.sort()` | list — ~в 2.8–3.9× быстрее, но это редкая одноразовая стоимость, не на каждую мутацию |
| Bootstrap `set_filter()` курсора vs list comprehension | list — ~в 3.7–5× быстрее, но это редкая одноразовая стоимость, не на каждую мутацию |
| Bootstrap `set_group()` курсора vs `list.sort(key=group)` | list — ~в 2.1–3.2× быстрее, но это редкая одноразовая стоимость, не на каждую мутацию |
